# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` — start Nuxt dev server on `http://localhost:3000`. **Do not run this yourself** — the user starts the dev server to test functionality.
- `npm run build` — production build.
- `npm run generate` — static site generation.
- `npm run preview` — preview production build locally.
- `npm run lint` — run ESLint and Stylelint together (both must pass).
- `npm run lint:fix` — auto-fix ESLint + Stylelint.
- `npm run lint:style` / `npm run lint:style:fix` — Stylelint only.
- `postinstall` runs `nuxt prepare` automatically; a Husky `pre-commit` hook runs `lint-staged` (ESLint on `*.{js,ts,vue}`, Stylelint on `*.{css,vue}`).

There is no test runner wired up despite `@nuxt/test-utils` being installed.

## Architecture

Single-page Nuxt 3 app (Vue 3, Composition API) that renders a 3D CSS cube with animated "sparkle" decorations on each face. The project is intentionally small; most interesting logic is CSS-driven.

- `app.vue` → `<NuxtPage />` → `pages/[[side]].vue` mounts a single `TheCube` with a `ProfileCard` in the `front` slot. The URL is the source of truth for the visible side: `/1`../`/6` map to `SIDES` positions (`utils/sides.ts`), `/` means `front`, anything else 404s via `validate`. Each spin does `router.push`, so browser back/forward replays rotations; the optional-param route keeps one component instance so rotations animate instead of remounting. The page background behind the cube is per-side (`SIDE_BACKGROUNDS` in `utils/sides.ts`: front `#9b509d`, back cyan `#4ee5ff`, the rest still share the dark `#241f42`) and crossfades over the same 2.1s as the spin via the `--page-background` custom property on `.main-page`.
- `components/TheCube.vue` is the central widget: a display-only component controlled via the `side` prop that renders six named slots (`front`/`back`/`left`/`right`/`top`/`bottom`) — the `left`/`right`/`top` faces each wrapped with a `SparklesSparkles` of a distinct `type`, while `front`/`back`/`bottom` carry dedicated ambience components instead (see MeteorShower/MeteorGlow, RippleField, GlitchField below). 3D transforms on `.cube[data-side="…"]` drive the rotation; hidden sides get `inert`, and sparkles are mounted only on the active side — constantly repainting edge-on (90°) layers otherwise rasterize as garbage-stripe GPU glitches in Chromium. (A `.bounds-plane` backdrop that pinned the scene's projected bounds for the same reason was removed once the cube got simple enough not to glitch — its 2% ink tint showed as a faint rectangle edge on the bright back-side page.) Rotation is triggered only by navigation — `SideNavigation` dots, the profile `Contacts` button, and arrow keys handled in `pages/[[side]].vue` (they navigate to the geometric neighbor from `SIDE_NEIGHBORS` in `utils/sides.ts`). There is no click-to-spin on the cube itself.
- `components/SparklesSparkles.vue` generates per-type sparkle particles. Positions come from a fixed `SPARKLE_POSITIONS_BASE` table + random offset, re-randomised on each CSS `animationiteration` via `updateSparklePosition`. Type-specific counts are clamped by `MAX_PARTICLES`. When modifying sparkle shapes, note that `star` renders an inline SVG while the others (`cross`/`plus`/`asterisk`/`diamond`) are built entirely from `::before`/`::after` pseudo-elements styled via `--sparkle-*` custom properties.
- `components/MeteorShower.vue` replaces sparkles entirely on the `front` (Profile) side: a page-space meteor rain mounted by `pages/[[side]].vue` as two viewport-covering layers — `layer="back"` swaps in for the mirrored background sparkles behind the cube, `layer="front"` sits after `TheCube` in the DOM and paints over it — so streaks pass both behind and in front of the cube (the `front` face itself renders no `SparklesSparkles`; the `right` face carries the `star` sparkles instead, and streak/head colors are hex derivatives of an acid mint `#38e897` chosen as the complementary contrast to the front side's pink accent `#df8eeb` — the head and tail tip stay saturated mint rather than fading to white). Streaks fly 25°±3° off vertical (down-left); tilt and translate derive from the same per-meteor angle so each flies along its own tail. All tuning lives in component constants: `LAYER_CONFIGS` (depth-correlated length/width/opacity/blur/travel/duration per layer), weighted spawn zones (mostly off-screen entries from the top/right edges — the zones must not overlap when projected perpendicular to the flight direction, or the double-fed corridor visibly overfills), and a `through`/`burnout` variant split (most meteors get travel stretched to exit the frame still lit; the rest dim out mid-sky). Respawns re-randomise lane/tilt on `animationiteration` but never touch `animation-duration`/`delay` — changing those mid-animation shifts the running phase. The long-term plan is a unique ambient animation per side.
- `components/MeteorGlow.vue` is the light those nearest meteors cast on the cube: mounted inside TheCube's front face after the slot (so the `mix-blend-mode: screen` patches brighten both the poster gradient and the ink typography), one soft elongated patch per front-layer streak. The front MeteorShower instance publishes its streak elements through `composables/useMeteorLightSources.ts` (a module-level `shallowRef`; retraction is owner-checked so a leave-transitioning old shower can't wipe a fresh instance's registration), and MeteorGlow drives the patches in a single rAF loop — all reads first (streak `getBoundingClientRect`, inline `--meteor-tilt`/`--meteor-length`, computed animated opacity), then compositor-only writes (`transform`/`opacity`). The head sits at rect center + (length/2)·(−sin θ, cos θ) (CSS clockwise rotation about the streak's center), the patch anchors backed off up the tail, and page coordinates map into face-local ones through the glow layer's own rect — an affine mapping valid only while the face is screen-parallel, so every frame first compares the projected layer rect against its layout size (±0.5px) and hides the patches mid-spin instead of smearing them over a rotating face — no fixed post-spin timer, and the settle moment doubles as the anchor of the fade-in ramp, which is multiplied into the JS-written opacity rather than transitioned on the layer (an ancestor `opacity < 1` would isolate the screen blending into a transparent group). Patch sizes are static per meteor and resolved once per publication; off-face patches are culled to `opacity: 0`. The shared mint base lives in the palette as `--color-meteor-mint`.
- `components/RippleField.vue` is the `back` ("How I work") side's ambience — the "wet print" concept: the side is a CMYK press sheet by acid water, with Cyberpunk 2077 / Marathon as the color reference. Unlike MeteorShower/GlitchField it splits `page`/`face`, not back/front: `layer="page"` is mounted by `pages/[[side]].vue` behind the cube (the acid-yellow backdrop water), `layer="face"` is mounted *inside TheCube's back face* (`v-if="side === 'back'"`) under the `WorkflowCard` slot content and is deliberately **still** — print-sheet furniture only (trim frame, crop marks, registration target, CMYK color bar, faint paper grain), no water, no filter, so the ink text reads undisturbed. The page scene is oversized (the bleed must exceed the warp's peak `scale/2` shift) and warped by a live SMIL-morphing `feTurbulence`+`feDisplacementMap` filter (`#ripple-warp-page`, scale 55–90 ≈ ±27–45px) — the honest heavy part, re-rasterizing every frame. The water itself is fully transparent — no colored wave overlays; the warp shows only through what it bends: a large magenta checkerboard (`.checker-sector`, `repeating-conic-gradient` of the side accent `--color-accent-back`) filling the top-right and bottom-left quadrants, each grid anchored to the corner touching the page center so only whole cells grow outward from the center (cells get cut by the screen edge alone) — regular straight strokes exist mostly to make the distortion legible, since displacement is invisible on flat color but obvious on a bent line. (Earlier iterations painted colored caustic webs, a giant outline-text echo of the headline, a registration crosshair and its circle's arcs over the water; all were removed in favor of the transparent-refraction checkers.) `WorkflowCard.vue` carries the matching typography: a Bungee `HOW I WORK` headline with misregistered cyan/magenta plates that slowly swim via animated `text-shadow` (the yellow plate is "invisible on yellow paper"), Space Grotesk principles each tagged with a plate swatch (cyan/magenta/key — yellow would vanish), and tracked-out caps for the tools line.
- `components/GlitchField.vue` is the `bottom` ("???") side's ambient animation, mounted by `pages/[[side]].vue` as the same two page-space layers as MeteorShower (the `bottom` face renders no `SparklesSparkles`). The back layer is dim purple lit strips; the front layer paints static scanlines, a pre-baked feTurbulence noise tile (motion faked via opacity/background-position jumps — the filter never re-runs), and glitch bars that corrupt whatever is painted beneath them via `backdrop-filter`. The front layer also rolls a single interference band over the whole scene at a constant CRT-like pace (painted before the bars so bar flashes corrupt it too). Bars stay `visibility: hidden` between flashes so their backdrop-filter surfaces only exist for the few visible frames; flash windows sit at the cycle tail and `animationiteration` respawns line/jitter only (never duration/delay). The front instance also schedules whole-frame displacement bursts (60–110ms every 2–4s): it re-seeds an SVG `feTurbulence`+`feDisplacementMap` filter def it owns and emits `frame-glitch`, and the page applies `filter: url("#glitch-frame-filter")` to `.main-page`, gated on `side === 'bottom'` so a burst never fires mid-spin. `LlmNoteCard.vue` carries matching RGB-split flashes on the title, terminal filename, and several terminal text fragments via a shared `.rgb-flash` utility (`content: attr(data-text)` pseudo clones; per-element cycle/phase/amplitude through `--rgb-flash-cycle`/`--rgb-flash-phase`/`--rgb-flash-shift`, plus a `.rgb-flash-stutter` modifier whose keyframes hit twice per cycle; splits are always horizontal, and targets must stay single-line for the clones to overlay — inline spans were chosen to fit the current line wrapping). It also hides a hacker-typer easter egg: printable keydowns while the side is active (guarded by checking the face's `inert`) reveal a canned tease at the blinking cursor, one character per keypress, Backspace erases.
- `components/ui/` (`Button`, `Text`, `Title`) are shared primitives consumed via Nuxt auto-import as `UiButton`, `UiText`, `UiTitle` (the `Ui` prefix comes from the `ui/` subfolder, per Nuxt's component-name convention). Each lifts itself one elevation level (`translateZ(var(--elevation-step))`) above its parent surface inside the cube's 3D context; `UiText` lifts arbitrary text content and renders its slot twice — the extra copy is projected one level down and repainted via `-webkit-text-fill-color` as a cast shadow; `UiTitle` is the pastel accent-tinted face heading built on `UiText`. The design is deliberately minimal: faces are flat solid `--surface-color` panels, there are no borders, gradients or filled cards (only the purple side's terminal keeps its own panel), and actions/links are bare text that answers hover with a pastel `--side-accent` tint plus an elevation lift. The `front` (Profile) face is the poster-style exception: its surface is a radial gradient from the accent `#df8eeb` at the center to `#964e9b` at the corners, typography sits on it in `--side-ink` (`#241f42`) heavy display caps (`--font-poster` Archivo Black) with condensed support lines (`--font-condensed` Oswald), and its actions hover to white instead of the pastel tint; its `--surface-color` is a `color-mix()` of the gradient endpoints (70% accent / 30% corner `#964e9b`) so `UiText` cast shadows keep a plausible local tone. The `back` (How I work) face is the second bright exception — the wet-print sheet: a deliberately uniform pale acid-yellow solid (`color-mix` of `--color-back-yellow` toward white, kept flat and still so the text stays readable) with `--side-ink` typography in the print pair (`--font-signage` Bungee, `--font-grotesk` Space Grotesk; all four families load via one Google Fonts link in `nuxt.config.ts`).
- `assets/css/global.css` is imported via `nuxt.config.ts` `css:` — `body` has `overflow: hidden` and a 500×500 minimum because the cube is a fixed-size viewport widget. Its `:root` block is the single source of truth for shared colors (`--color-*`) and the font pairs (`--font-*`): every shared color is defined exactly once there; consumers reference the var or derive alpha/tint variants via CSS relative color syntax (`rgb(from …)`) and `color-mix()` — plain `#fff` is the only color kept inline. The `SIDE_COLORS`/`SIDE_BACKGROUNDS` maps in `utils/sides.ts` hold `var(--color-…)` references, not hex.

There is **no FSD structure in use** (commit `9a11a3e removed FSD` explicitly reverted it). Do not reintroduce `app/widgets/features/entities/shared/` directories unless the user asks — put new work under `components/`, `pages/`, etc.

## Code style specifics (beyond the user-global rules in `~/.claude/CLAUDE.md`)

The ESLint + Stylelint configs here are strict and opinionated. Run `npm run lint` after edits; do not manually reformat against these rules if lint will auto-fix them.

- **TypeScript**: `tsconfig.json` enables `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noPropertyAccessFromIndexSignature`, and `noImplicitOverride`. Array/record accesses return `T | undefined` — narrow before use (see `TheCube.vue` guards on `nextSide`).
- **ESLint**: `@typescript-eslint/explicit-function-return-type` is on — always annotate return types on functions. `no-explicit-any` is an error (use `unknown`). `consistent-type-definitions` requires `interface` over `type` for object shapes. `padding-line-between-statements` enforces blank lines around `const`/`let`/`return`/control-flow blocks — most whitespace "weirdness" you see is mandated.
- **Vue template rules**: `component-name-in-template-casing: PascalCase`, `attribute-hyphenation: always`, strict attribute ordering (`DEFINITION → LIST_RENDERING → CONDITIONALS → … → EVENTS`), and `first-attribute-linebreak: below` for multiline elements.
- **Stylelint `order/properties-order`**: CSS properties must follow a fixed order (positioning → box model → typography → background → border → effects → animation → cursor → content). See `.stylelintrc.json` for the full list. Also: `color-named: never` (use hex), `declaration-no-important: true`, strings use double quotes.
